COMOS Walkinside APIs
API Compatibility Policy

This is part of general COMOS Walkinside Compatibility Policy, concerned specifically with API compatibility.

Goal of this document: to establish clear rules on what customers can expect in terms of compatibility and what minimum compatibility R&D commits to maintain.

Applies to: official standard releases of COMOS Walkinside.

This document mentions 2 directions of compatibility:

  • Backward compatibility: refers to situation when software is combined with an older component or data than itself.

    Example: a SimCity binary released for Windows 95 is launched on Windows 7. If it runs correctly, Windows 7 is said to be backward compatible with Windows 95 with respect to SimCity.

  • Forward compatibility: refers to situation when software is combined with a more recent component or data than itself.

    Example: a document created by Microsoft Word 2010 is opened with Microsoft Word 2007. If it opens correctly, Word 2007 is said to be forward compatible with Word 2010 with respect to document format.

API Compatibility

API compatibility refers to the ability of API client, which was created for some API release X, to be run on API release X', where X' is more recent or less recent than X. For example, creating a viewer plug-in or a builder for 7.0 and running it in either 10.2 or 6.2.

For APIs the following kinds of compatibility are distinguished:

  • Binary compatibility: a client component built against one release of API works with another release of API without being rebuilt.
  • Source compatibility: a client component built against one release of API works with another release of API if it is rebuilt against that release.
  • No compatibility: a client component built against one release of API does not work with another release of API; rebuilding it requires modification of component's source code.

Technical note: it is possible to have binary compatibility, but not source compatibility, and vice versa. One example is C# named parameters: renaming parameters is a backward-compatible, but source-incompatible change.

API Support Levels

This policy establishes different API support levels:

  • None
  • Preview
  • Official
  • Deprecated
  • Discontinued

"Preview" APIs do not get any compatibility guarantees. "Best effort" is provided to avoid gratuitous incompatibility.

"Official" APIs get higher compatibility guarantees:

  • Full backward compatibility with SP0 release:

    Each new Major.Minor.SPX release maintains binary compatibility relative to its Major.Minor.SP0 release. See examples below.

  • Limited backward compatibility with previous SP0 release:

    Each new Major.Minor.SPX release maintains binary compatibility relative to Prev(Major.Minor).SP0 (its "previous" SP0 release), except for API members that were deprecated by any Prev(Major.Minor).SPX. Clients that used such deprecated API members may become broken; clients that did not use them are not affected.

  • Removal of deprecated API members only in SP0 releases:

    Any Major.Minor.SP0 release may remove or change any API members that were marked as deprecated in any Prev(Major.Minor).SPX release. If this happens, compatibility is broken for clients that use those deprecated members. "Best effort" is provided to avoid gratuitous breakage of compatibility. If a deprecated member is not changed in this Major.Minor.SP0 release, it must stay compatible in all following Major.Minor.SPX releases at least until Next(Major.Minor).SP0.

  • Friendly deprecation of API members:

    Any release of at least Service Pack level may mark any API members as deprecated. "Best effort" is provided to point user to an alternative non-deprecated way to achieve same functionality as the one being deprecated.

  • Source compatibility is maintained on a "best effort" basis, but is not guaranteed.

API that becomes "official" in some release does not have to maintain compatibility with previous releases, as they were just "preview".

"Deprecated" APIs are official APIs that are being phased out. Normally there is some alternative API to achieve same tasks. Deprecated APIs can be expected to be discontinued in the closest SP0 release. If you rely on an API that has been deprecated and you do not find a good alternative

  • please raise your concerns at https://github.com/walkinside/sdk/issues or otherwise contact us.

"Discontinued" APIs do not get any guarantees or support and are not actually considered as APIs anymore.

Like any complex software, Walkinside has a number of internal interfaces between components. Those are not considered APIs and their use by 3rd-parties technically is reverse engineering, so is unsupported (support level "None"). These are some interfaces known to be internal:

  • Viewer SDK's JavaScript interface;
  • Server's web-service interface;
  • any events passed via Viewer SDK's IVREventManager;
  • any public members placed in *.Private.* namespace of SDK assemblies;
  • any public members outside of SDK assemblies.

Recognized Walkinside APIs

As of version 10.2, Walkinside provides several APIs:

  • Viewer SDK API: intended for developing Walkinside plug-ins and alternative Walkinside viewer-like applications.
  • Data SDK API: intended for accessing Walkinside project data in data-centric and non-interactive scenarios.
  • Core SDK API: intended to augment Viewer SDK API with custom model rendering capabilities.
  • Virtual Room Client Library API: intended for developing Virtual Room agents that can observe other participants, drive avatars and communicate via custom messages.
  • Viewer SDK HTTP Automation API: local HTTP endpoint intended to give HTML-based applications a limited way to drive Walkinside camera via hyperlinks.
  • CLI (command-line interfaces): intended for batch scripting and automation. Provided by the following applications:
    • Viewer
    • Builder for PDMS, batch mode
    • Builder Tool
    • MicroStation Exporter plug-in, batch mode
    • PDS Exporter plug-in, batch mode
    • SP3D Extractor
    • Project Data Transfer Tool
    • Portal Client Tool
    • Virtual Room Service

Examples

Below are examples of deprecation and removal of API members. For the purpose of examples, this sequence of releases is used:

  • 5.4
  • 6.0 (version 6.0.2007.0)
  • 6.1 (version 6.1.2214.0)
  • 6.2 (version 6.2.24.16.0)
  • 7.0 (version 7.0.26.26.0)
  • 7.0 SP1 (version 7.0.28.23.0)
  • 7.0 SP2 (version 7.0.29.16.0)
  • 7.0 SP3 (version 7.0.30.32.0)
  • 10.2 (version 10.2.32.77.0)

Example 1.

Walkinside 6.1 marks IVRSpam member of Viewer SDK as deprecated. It offers IVRBacon as alternative for some of IVRSpam functionality and just drops remaining functionality. Viewer SDK plug-ins are recommended to stop using IVRSpam.

Walkinside 6.2 and 7.0 do not have to, but still keep IVRSpam unchanged, so it stays compatible.

Because 7.0 kept those deprecated APIs, all 7.0 service packs (SP1, SP2, SP3) must keep IVRSpam compatible.

Walkinside 10.2 decides to drop IVRSpam, at which point it is just removed from API.

Viewer SDK plug-ins that used IVRSpam are forced to be updated to use IVRBacon and to be rebuilt and redeployed.

Example 2.

Walkinside 7.0 SP3 marks IVRSpam member of Viewer SDK as deprecated.

Any Walkinside 7.0 service packs and updates coming after SP3 must keep IVRSpam compatible.

Walkinside 10.2, as a new SP0 release, decides to drop IVRSpam, at which point it is removed from API.

Example 3.

Walkinside 6.0 adds new member VRHam to .NET bindings of Exporter SDK API.

Walkinside 7.0 SP3 decides VRHam is not useful anymore and removes it. This breaks compatibility for VRHam users, but no deprecation or alternative is required, because Exporter SDK API is in "preview" status.

API Versioning

APIs follow semantic versioning scheme (http://semver.org) to indicate to clients when there is compatibility breakage.

For example, removing deprecated item from "official" API means that API version must jump to the next major version number. API versions necessarily differ from software versions.
